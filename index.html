Literature: Reading and Management

今天將Mendeley裡面的參考文獻從700多篇刪減到了200多篇，現在看著Mendeley果然爽多啦！

到目前為止，嘗試過很多種參考文獻管理軟體。目前只用兩個：Mendeley和Zotero。然而，兩者的差別還是很大。因此，兩種軟體算是各有分工啦！

Mendeley：經典文獻，用於跨平台閱讀

由於Mendeley可以很好地同步到網路端，並且在線閱讀體驗非常好。此外，根據Mendeley上的某篇文獻，可以查閱類似的文獻。因此，非常方便進行文獻擴展以及和不同學者協作。

因此，Mendeley適合僅僅保存經典文獻，在laptop、網頁端、iPad等多處進行同步，并進行隨時閱讀。但是，切記只能保存經典文獻，否則你會深刻理解什麼是“資訊過載”。

Zotero：所有文獻，用於論文寫作

通過Chrome或者Firefox的Zotero插件，可以一鍵保存參考文獻。因此，可以針對每個project建立一個文件夾。然後，將文件夾導出為.bib文件，同時可以根據自動生成的citation key在Latex或者Markdown中進行寫作。

所以，Zotero適合保存所有的文獻，并根據不同的project進行分類。當然，同一篇文獻可以在不同的project中，但是在Zotero中只會有一個存檔。

PDF閱讀器

很難找到合適的PDF閱讀器。就目前的嘗試來看，有兩款是十分適合的：Adobe Acrobat，以及Utopia Documents。後者很多功能尚待嘗試。

引用文獻格式

在引用文獻的處理上，也是學術論文寫作中比較麻煩的事情。在不同情形下，對於引用文獻格式的要求差異較大，因此也要分別處置。

文獻管理軟體

個人使用過多種文獻管理軟體，例如Endnote，Readcube，Mendeley，Zotero，Jabref等。最後依然一直在使用的只有兩個：Zotero和Mendeley。Endnote有多麼非人道自不必說，Readcube的免費版本畢竟功能太少，而Jabref的界面總是覺得不舒服。

現在一直在使用Zotero和Mendeley，並且兩者各有其分工。

  • Zotero: 就寫作的方便程度而言，感覺還是Zotero比較實用一些。我的一般做法是：在查找文獻的階段，為每篇論文設置一個folder，從Chrome瀏覽器上一鍵保存文獻到此folder；在寫作時文中引用的階段，使用快捷鍵獲取citation key，然後插入到論文文稿中；而在文末生成文獻的階段，則將此folder中的所有文獻批量導出為.bib格式的文件。這樣，也就完成了全過程。
  • Mendeley: Mendeley則十分方便查閱同行的文獻閱讀與引用狀況，並且十分方便進行在線文獻閱讀。例如，我沒有帶自己的laptop，然後在別人的辦公室要呆一下午。那麼，這個時候就可以登陸網頁版的mendeley進行文獻閱讀。近期mendeley進行改進之後，在線閱讀體驗已經非常棒了。mendeley還有一個很讚的功能，即是文獻共享。假設你與A君共同完成一篇論文，那麼可以創建一個folder，將需要的文獻放入，并邀請A君加入。這樣可以方便進行文件共享和討論。

Latex文獻引用方案

在Latex中進行寫作時，十分方便。先在zotero中生成citation key，從而在TeXStudio中進行論文寫作。最後將zotero中所有相關文獻導出為.bib格式的文檔，利用Latex宏包（諸如apacite，natbib等）設置，便可以生成需要的文內引用和文末顯示格式。

其它文獻引用方案

當需要將論文文稿轉為Word，或者直接在Pandoc Markdown中進行寫作時，就要考慮文獻引用方案了。此時便要複雜一些，具體的方案是：使用CSL（citation style language）。以下具體進行說明。

CSL：Citation Style Language

協同使用CSL和Pandoc，可以方便使用Markdown寫作，并將其轉換為需要的格式。對於CSL而言，現有上千個格式可供下載，也可以自己修改csl格式。具體可參閱以下網頁：

  • CSL介紹：請參閱https://www.zotero.org/support/zh/dev/citation_styles。
  • 修改CSL：請參閱https://www.zotero.org/support/zh/dev/citation_styles/style_editing_step-by-step。

對於大部分引文格式，都可以從zotero上直接找到。例如，簡體中文中常出現的格式，即可通過zotero提供的數字標註法的引文格式列表中找到簡體中文參考文獻標準格式并下載即可。

對於使用LaTeX的情形，則需要使用GB/T 7714-2005文獻著述標準對應的.bst文件。具體可參閱說明網頁以及在GitHub上的倉庫。

設置好引文格式csl之後，并保存為mystyle.csl格式的文件。然後在Pandoc命令中使用：

--csl mystyle.csl

來進行最後的文獻生成。

Scholarly-oriented Markdown

Markdown十分輕巧，對於部落格、隨筆等一般要求的文章已經十分方便。然而，對於學術寫作而言，則使用起來其功能未免捉襟見肘。若是使用Latex的話，自然是足夠了，但是Latex寫作（特別是滿屏幕的反斜杠）卻又不是很人道。況且有很多時候某些雜誌社非要提供Word格式的稿件，即便用Pandoc將Latex文檔轉為Word，也還是難免需要調整半天格式。因此，出於學術寫作目的的Markdown解決方案還有待觀察。

Pandoc Markdown

Pandoc提供了多種格式之間相互轉換的方案。此外，Pandoc還提供了多種Markdown的擴展，以形成了Pandoc Markdown。其語法的中文翻譯版可參閱：http://pages.tzengyuxio.me/pandoc/。另外一份的寫作說明可參閱：http://pandoc.herokuapp.com/。其中顯著的特點是對參考文獻以及數學公式進行了擴充，從而使之有取代Latex的可能。然而，其解決方案依然並不理想。

Scholdoc: A solution for Scholaryly Markdown

Scholdoc頗具雄心地提出了一套解決方案，也看起來十分具有前景。具體可參閱GitHub上的專案：https://github.com/timtylin/scholdoc。不過目前還在測試階段，預計將在2015年秋季發佈。

回歸寫作

久違了寫作的感覺。如今我用Hexo+GitHub+Markdown組合搭建了部落格，準備重新回歸寫作啦。

數年前，正值讀書和寫作的密集期，時常泡上一杯茶，安靜的坐在電腦旁，任由思緒從指間流過，藉由鍵盤而顯現在屏幕上。那時的自己，喜與人辯論，而常以士子與報人自任。士子當“觀乎人文以化成天下”，而報人則意在“轉移一時之風氣”。士子與報人，皆是關乎改變外界環境，是“外王”之道的踐行者。那時與十來個朋友，在後山討論一些無關自身的問題，或是兩宋經濟中心轉移，或是明末平遼方略。也曾商議過在校內辦報的事宜，卻恰逢輿論鉗制加強而作罷。如今雖已經各奔東西，但後山不鳴不飛亭想必不會忘記我們的少年意氣。

“輕飄飄的舊時光就這麼溜走，轉頭回去看看時已匆匆數年。”旅行和攝影，變成了近幾年的興趣。旅遊的意義至少有兩點：開闊胸襟與通達物情。在烏珠穆沁草原上策馬奔馳，感受著迎面而來的涼風，朝著看不著盡頭的遠方而去，等到回首之時牧馬人已變成了一個小點而已。大抵在草原上才能真正充分體驗到自由的感覺，煩惱在那時消失不見，只感到無盡的暢快，感到那融入天地之間的欣喜。很難解釋自己為什麼那麼喜歡草原，或許獅子本就是屬於草原的吧。在太平洋岸邊騎行的時候，若是風平浪靜，則很難分辨出蔚藍色的大海與天空的分界。偶爾也會遇到驚濤拍岸，捲起幾丈高的海浪，尤為壯觀。如此景觀，會讓我們意識到自己的有限與渺小，也自此更能心胸開闊地接受許多事情。在遊歷中為大自然的壯麗所震撼，胸襟便日漸開闊而更具接納能力了。

通達物情最好的例子莫過於“濠梁觀魚”。莊子與惠子游於濠梁之上，莊子曰：“鰷魚出遊從容，是魚之樂也”。魚游於水，鳥棲於陸，雖是不同的個體，但各率其性，皆達逍遙之境。這便是莊子通達物情，所以能體會到魚兒出遊之樂。在旅遊的過程中不必考慮什麼瑣事。輕裝上路，因而更能觀察和體驗到平時所不能體驗到的微妙。在徹底放鬆的身體狀況下，不斷探索旅途中的未知與新奇，自然調動了我們的敏銳和細膩。在此情境下，我們更有可能通達物情，從而與大自然交流並獲得新的感悟。“天行健，君子以自強不息；地勢坤，君子以厚德載物”，這句話是昔人格物致知的絕佳範例：觀察并思索周圍事物的“德性”，以資修身。由體察天的“健”德到樹立人的“健”德，因而推出君子當自強不息。濠梁觀魚，也可藉由魚兒的逍遙來培養自己的通透無礙。

在過去的五年裡面，幾乎沒有寫過任何東西。在閱讀和遊歷的過程中，接觸到了更多的思維體系和生活方式，使得自己變得愈來愈謙遜。不再那麼不知天高地厚，因而也再難寫出什麼了。畢竟，該說的，前人大抵都已經說過。所以昔時喜讀經史，而今卻偏愛各種好玩的雜集了。讀過些西方哲學與諸子之說，也就不想聽當世之人講什麼大道理了。反倒更願意聽聽不同個體的體驗與故事。去年讀心斎之《幽夢影》，便是體驗居多。如“痛可忍而癢不可忍，苦可耐而酸不可耐”，我便深有體會。作者與友人之間的相互評論打趣，簡直就是前清的微信朋友圈、Facebook、微博等，感覺特別好玩！近來偶爾翻翻《世說新語》，也簡直是後漢魏晉時代的逗比全集！想想熟識的朋友中也不乏此類，也是醉了。

所以，而今重新回歸寫作，記錄些好玩的事情、特別的體驗。多年以後，當許多事情在記憶中不復存在時，若能在而今留下的這些文字中發現些值得賞玩的過往，想必是極好的。

使用Pandoc將LaTeX文檔轉化為Word

Pandoc 可以在不同的格式之間進行相互轉換，詳見http://johnmacfarlane.net/pandoc/。因此可以使用Pandoc Markdown進行論文寫作，并通過Pandoc將其轉化為.tex或者.docx文檔。

然而，有時會直接進行LaTeX寫作，而某些期刊卻又要求投稿時使用Word格式文檔。這時頭痛的事情便來了。當然，也可以用Pandoc來進行初期的轉化工作啦！

大致而言，將.tex格式的源文檔轉化為 .docx格式，可以採用以下命令行：

pandoc -s -o manuscript.docx manuscript.tex

由於.tex文件中大多使用.eps格式的圖片，因此導致在Word文檔中對應的圖片無法顯示。因此可以採用如下命令行設置圖片格式：

pandoc -s -o manuscript.docx manuscript.tex --default-image-extension=png

接下來，便是參考文獻的問題了。我通常為每個論文收稿都建立一個zotero分類，并將其導出成為.bib格式的參考文獻。可以採取如下方式自動將.bib格式的參考文獻以及.tex文檔一起，轉化為Word格式，并保留完整的參考文獻文中引用及文末列出。當然，還需要下載APA格式對應的apa.csl文件啦：

pandoc -s -o manuscript.docx manuscript.tex --biblio ref.bib --csl apa.csl --default-image-extension=png

我們還可以事先在一個空白的Word文檔中設置好正文、一級標題、二級標題、單欄雙欄等格式，命名為template.docx。然後按照以下命令行生成對應格式的Word：

pandoc -s -o manuscript.docx manuscript.tex --biblio ref.bib --csl apa.csl --default-image-extension=png --reference-docx=template.docx

不過估計後續還是需要手動調整Word文檔的格式。万恶的Word啊，簡直是醉了！

Getting Things Done: 技術宅如何治療拖延癥？

相信很多人與我一樣，都有著嚴重的拖延癥。那麼，有三大利器可以幫助治療拖延癥： Trello, RememberTheMilk,以及Google Calendar。

Trello

Trello非常適合用於專案管理(Project Management)，根據其設置共分為四個層級：

  • Board: 每個賬號下可以管理多個Board，一般而言，一個專案對應一個Board。
  • List: 可以進行專案的流程劃分，或者劃分為子專案。例如在進行某個專案時，可以採取“開發-測試-整合”等流程設置List。
  • Card: 根據任務設置Card，並且通過Member和Deadline的功能來進行協作以及進度控制。
  • CheckList: 具體事項。

此外，Trello也非常適合多人協作。在Trello中，有以下協作方式：

  • Member: 在每個Board或者Card當中，都可以加入協作者。
  • Activity: 對於每個Card，可以通過附件attachment以及評論來進行具體任務的討論。

Remember The Milk

當在研究所裡面的時候，可以使用RememberTheMilk每天對任務進行打印。這樣非常方便進度的管理。

當然，我個人喜歡在Lab裡面使用它。此外，和Trello配合使用也是非常讚的！具體的做法是：

  • 在Trello中進行長期的專案管理。例如，處理半年或者更長時間的各項專案。
  • 在Card層級上，亦即任務層級上，配合使用RememberTheMilk使用。每天打印出具體的Card。具體到每天，具體到每個任務(Card)。
  • 在每個任務下面的具體事項(Tasks, CheckList)，依然採用Trello進行管理。

Google Calendar

最喜歡的功能是SMS提醒。當不在研究所或者辦公室的時候，可以讓Google Calendar傳短訊來提醒你該做什麼！當然，則需要手機號碼通過Google的短訊驗證，並且經過簡單的設置就可以啦。曾經有好長一段時間，我都是每天晚上寫好第二天的日程，然後讓Google Calendar發短訊來提醒我什麼時候應該去什麼地方做什麼事情，想想也是醉了！

样本容量与模型验证

模型验证的方法

模型验证有两种方法：

  • 如果在某个数据集上发现了某种模式，那么就用其它的数据集来验证它
  • 用概率论来测试发现的模式是否是巧合

前者在统计学习中十分常见，即先用训练集来发现模式，并用测试集来验证；后者则是统计学中假设检验的思路，通过诸如p值等指标来判断发现的模式是否是随机的。

在通常情形下，总体样本非常大，数据收集非常困难。因此，统计学者通过抽样设计来从总体中选择样本进行研究。常见的方式是随机抽样，而后续的研究也奠基于这个假设。由于数据收集成本较高，研究样本也相应较小。在此情形下，研究者采取第二种方式，通过构建模型、提出研究假设来描述对数据模式的先验判断，并进行假设检验来验证。先验判断一般基于经济学理论，辅以必要的逻辑推理。这种研究方法适宜于研究样本较少的情形，其缺点在于过渡拟合（overfitting），亦即将随机因素错误地识别为某种规律，或者说将噪音错误地识别为信号。

当研究样本较大时，则可以将数据集随机分为训练集和测试集。首先试图在训练集中发现某种模式，然后通过交叉验证等方式在测试集上观测效果。训练集可能会产生多种模式，而测试集上的效果这有助于选择最佳的模式。这种研究方法适宜于研究样本较多的情形，否则将数据集进一步缩小将无法发现真实的模式（如果存在的话）。

大数据改变了什么？

技术进步使得观测数据的获取变得更加容易。当然，收集访谈数据、问卷数据的成本并未显著降低。因此，谈论大数据时，更多地是在谈论观测数据，且大部分存在于企业的数据库中。

从统计上看，样本容量有显著变化：大数据情形下，样本容量剧增，甚至可以获取总体数据。在此极端情形下，基于抽样理论的第二种模型验证方法则不再适用。因此，需要转而考虑的是基于训练—测试思路的第一种模型验证方法。因此，在考虑PlantRetail上数据集时，可能更加适合采用第一种模型验证方法。显然，此时研究总体为大型零售商而非所有零售商。如此定义研究总体的话，PlantRetail上还不是总体数据，因此第二种模型验证方法也同样适用。所以可以将二者进行比较，或许有些有趣的发现。
