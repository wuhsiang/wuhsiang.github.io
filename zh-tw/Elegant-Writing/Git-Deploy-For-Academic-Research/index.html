使用GitHub託管學術論文的若干設置

採用GitHub的private repository來託管正在進行的學術論文是絕佳的選擇，這樣每次的更改都會被記錄，而且還可以通過GitHub的記錄來查看自己有沒有偷懶哦！

Git流程

關於Git使用流程，較好的教程有：

  • GitHub Flow
  • A successful Git branching model
  • 一個成功的Git分支模型

存在三種分支：主分支(master)，開發分支(develop)，和支援分支(feature, release, hotfix)。

大致說來，始終存在的兩個分支是：

  • master: 當develop分支上的代碼達到穩定的狀態時，就可以合併回master分支，從而發佈版本。因此，每次有變化被合併到master分支時，根據定義就是一次新的產品版本發佈。
  • develop: 反應開發的進程。

此外，還有一些有限壽命的分支，這些分支被稱為支援分支，且最終都會被合併入develop或者master分支之後，再被刪除。支援分支主要分為：

  • 特性分支(feature branch)： 也被成為topic分支，源自develop分支，而最終也被合併回develop分支。feature branch為即將發佈的版本開發新功能。
  • 發佈分支(release branch)：源自develop分支，而最終被合併回master和develop分支。release branch為新的產品版本發佈提供準備和支持。一般命名為：release-*。
  • 熱補丁分支(hotfix branch)：源自master分支，而最終被合併回master和develop分支。hotfix branch用於處理未預期的重大bug時，進行快速的產品bug修復。一般命名為：hotfix-*。

學術論文的Git寫作流程

由於涉及正在進行的研究，因此需要付費購買私有倉庫(private repository)。最理想的情形自然是為每個project建立單獨的private repository。由於已經完成的、正在進行的、打算進行的研究加起來會比較多，因此可能需要採用折中的方案。個人使用的是micro plan，也就是可以最多建立5個私有倉庫，每年支付84美元。因此，我並沒有為每個project建立單獨的private repository，而是將類似主題的研究全部放到一個private repository裡面。有時候，一個私有倉庫裡面可能有四五個具體的研究專案。這時候，就需要有寫作流程的計劃了。

參照一般的Git開發流程，我採用了以下方案：

  • master branch: 用於保存穩定的研究專案狀態。換言之，用於保存已經完成初稿並且已經向期刊投稿了的研究專案。
  • develop branch: 用於保存正在進行的研究專案。一旦研究已經完成並且投稿，則將其併入到master branch。
  • feature branch: 用於某個特定的研究專案寫作。例如在private repository裡面同時有五個具體的研究專案，那麼可以分別從develop branch當中分離出來成為單獨的feature branch，並且命名為topic-*。例如在研究dynamic lot sizing的時候，我使用了如下特性分支： topic-dls-learning, topic-dls-deterioration, topic-dls-repeatpurchase等等。
  • release branch: 較少使用。
  • hotfix branch: 用於已經投稿的稿件的後續修改。例如，當收到審稿意見之後，可能進行的修改；或者論文拒稿之後的修改和重投等等事宜。這樣處理的邏輯在於：已經投稿的研究專案都已經保存在master branch當中，而hotfix是從master當中分離，並且最後會合併回master和develop分支的。因此，hotfix的功能很適合用於已投稿件的後續修改。

設置忽略文件

在使用GitHub的時候，也需要進行一些特殊的設置。

原則上講，所有中間過程中生成的臨時文件都最好被忽略掉。例如，在使用LaTeX寫作的時候，就會生成很多臨時文件。

忽略的文件類型可以在.gitignore中進行設置，具體方法可參閱：https://help.github.com/articles/ignoring-files/。由於在windows下無法直接創建沒有文件名的.gitignore文件，所以需要在Git Bash中進入專案目錄，并通過以下命令行創建：

touch .gitignore

然後，可以發現新建了一個.gitignore文件。接下來可以在文件中進行如下設置：

.aux .blg
.dvi .log
.synctex.gz -eps-converted-to.pdf

於是，就可以忽略LaTeX的臨時文件啦。

此外，可能很多臨時文件已經被跟蹤。此時，可以採用命令從repository中刪除臨時文件，如：

git rm -f *.dvi

更多詳細內容可以參見Git基礎。
